# TASK.md

## Task Name
Spawn player near assigned base

## Goal
Игрок должен появляться не на стандартном `SpawnLocation`, а рядом со своей назначенной базой.

## Context
- Базы уже существуют и назначаются через `BaseService`
- Нужно изменить только точку появления игрока
- Остальную архитектуру не менять

## Scope

### In scope
- Телепорт игрока к своей базе после назначения базы
- Телепорт игрока к базе при респавне
- Fallback, если база ещё не назначена

### Out of scope
- Изменение системы баз
- Изменение UI
- Изменение сохранения данных
- Изменение механик геймплея

## Files Allowed to Change
- `src/server/Modules/PlayerService.lua`
- `src/server/Modules/BaseService.lua`
- `Workspace` объекты баз, если нужен `SpawnPoint` или аналогичный part

## Related Contracts
- `docs/ARCHITECTURE.md`
- `docs/systems/BASES.md`
- `docs/contracts/API_RULES.md`

## Expected Behavior
- После входа игрок оказывается рядом со своей базой
- При смерти и респавне игрок снова появляется возле своей базы
- Если база ещё не назначена, используется fallback spawn
- Решение работает стабильно и не ломает другие системы

## Recommended Implementation Notes
- Лучше хранить точку спавна внутри каждой базы, например `BaseN/SpawnPoint`
- `BaseService` должен уметь возвращать spawn-локацию для игрока
- `PlayerService` должен применять spawn после загрузки профиля и после `CharacterAdded`
- Использовать серверную логику, без client-side teleport

## Acceptance Criteria
- Игрок не появляется на общем `SpawnLocation`
- Игрок появляется рядом со своей базой
- После респавна игрок снова оказывается у своей базы
- Если база не назначена, игрок не ломается и получает fallback spawn
- В Output нет ошибок

## Verification Steps
1. Запустить игру с 1 игроком
2. Проверить, что персонаж появляется возле своей базы
3. Умереть и проверить респавн
4. Проверить, что респавн тоже происходит возле базы
5. Проверить поведение, если база ещё не успела назначиться
